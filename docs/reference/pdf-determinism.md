# PDF Determinism Guarantees

This document describes the determinism guarantees for PDF output generated by rtfkit's Typst-based renderer.

## Overview

rtfkit's PDF renderer is designed to produce **byte-identical output** for the same input when using deterministic options. This is critical for:

- **Reproducible builds**: Same source always produces same PDF
- **Version control**: PDFs can be diffed and tracked meaningfully
- **CI/CD pipelines**: Consistent output across different build environments
- **Caching**: Output can be cached reliably

## What is Guaranteed Deterministic

### Content and Layout

| Aspect | Guarantee |
|--------|-----------|
| Text content | ✅ Identical across runs |
| Text formatting (bold, italic, underline) | ✅ Identical across runs |
| Paragraph alignment | ✅ Identical across runs |
| List structure and numbering | ✅ Identical across runs |
| Table structure and cell content | ✅ Identical across runs |
| Page breaks | ✅ Identical across runs |

### PDF Metadata (with fixed timestamp)

| Aspect | Guarantee |
|--------|-----------|
| Creation date | ✅ Fixed when `fixed_timestamp` is set |
| Modification date | ✅ Fixed when `fixed_timestamp` is set |
| Creator field | ✅ Always "Typst {version}" (deterministic per version) |
| Document identifier | ✅ Derived from content hash |

### Font Handling

| Aspect | Guarantee |
|--------|-----------|
| Embedded fonts | ✅ Always used (no system fonts) |
| Font resolution order | ✅ Consistent (embedded fonts only) |
| Font fallback chain | ✅ Deterministic (Typst embedded fonts) |

## How to Enable Deterministic Output

### Using the Library

```rust
use rtfkit_render_typst::{document_to_pdf_with_warnings, RenderOptions, DeterminismOptions};

let options = RenderOptions {
    determinism: DeterminismOptions {
        fixed_timestamp: Some("2024-01-01T00:00:00Z".to_string()),
        normalize_metadata: true,
    },
    ..Default::default()
};

let output = document_to_pdf_with_warnings(&doc, &options)?;
```

### Using the CLI

```bash
# Deterministic output with fixed timestamp
rtfkit convert input.rtf --to pdf --fixed-timestamp "2024-01-01T00:00:00Z" -o output.pdf
```

## Determinism Options

### `fixed_timestamp`

When set, this ISO 8601 timestamp is used for:
- PDF creation/modification date metadata
- Document date (if not explicitly set in source)

**Format**: ISO 8601 (e.g., `"2024-01-01T00:00:00Z"`)

**Default**: `None` (uses current time)

### `normalize_metadata`

When `true`, normalizes metadata for deterministic output:
- Ensures consistent field ordering
- Applies fixed timestamp to all date fields

**Default**: `false`

## Environment Independence

The PDF output is **independent of the runtime environment**:

| Environment Factor | Impact on Output |
|-------------------|------------------|
| System fonts | ❌ Not used (embedded fonts only) |
| PATH variable | ❌ Not used (in-process rendering) |
| Current directory | ❌ Not used (in-memory processing) |
| System time | ❌ Not used (when `fixed_timestamp` set) |
| External binaries | ❌ Not required (no shell-out) |
| Network access | ❌ Not required (offline capable) |

## Verification

### Automated Tests

The determinism test suite verifies byte-identical output:

```bash
# Run determinism tests
cargo test --package rtfkit-render-typst --test determinism_tests
```

Test categories include:
- Basic content (paragraphs, lists, tables)
- Nested structures
- Unicode content
- Edge cases (empty documents, special characters)
- Environment independence

### Manual Verification

To manually verify determinism:

```bash
# Generate PDF twice with fixed timestamp
rtfkit convert input.rtf --to pdf --fixed-timestamp "2024-01-01T00:00:00Z" -o run1.pdf
rtfkit convert input.rtf --to pdf --fixed-timestamp "2024-01-01T00:00:00Z" -o run2.pdf

# Compare outputs (should be identical)
diff run1.pdf run2.pdf && echo "Deterministic!"
```

## Implementation Details

### Typst Integration

The renderer uses the Typst library for in-process PDF generation:

1. **IR → Typst Source**: rtfkit IR is converted to Typst markup
2. **Typst Compilation**: Typst compiles the markup in-process
3. **PDF Export**: Typst exports to PDF with controlled options

### Timestamp Handling

The fixed timestamp is passed through two paths:

1. **Typst World**: The `today()` method returns the fixed date
2. **PDF Options**: The `PdfOptions.timestamp` field sets PDF metadata

```rust
// Convert Unix timestamp to Typst PDF Timestamp
fn unix_to_typst_timestamp(unix_ts: i64) -> Option<typst_pdf::Timestamp> {
    let datetime = chrono::DateTime::from_timestamp(unix_ts, 0)?;
    let typst_datetime = Datetime::from_ymd_hms(
        datetime.year(),
        datetime.month() as u8,
        datetime.day() as u8,
        datetime.hour() as u8,
        datetime.minute() as u8,
        datetime.second() as u8,
    )?;
    Some(typst_pdf::Timestamp::new_utc(typst_datetime))
}
```

### Font Embedding

Fonts are embedded from `typst-kit` with the `embed-fonts` feature:

```rust
// Uses embedded fonts only - no system fonts
let fonts = typst_kit::fonts::FontSearcher::new()
    .include_system_fonts(false)  // Critical for determinism
    .search();
```

## Known Limitations

### Typst Version Dependency

The PDF output includes the Typst version in the Creator metadata:
```
Creator: Typst 0.14.2
```

This means:
- Output may differ between Typst versions
- Output is deterministic for a given Typst version
- Pin your `typst` dependency version for reproducibility

### PDF Internal Structure

While the visible content is deterministic, internal PDF structure may vary:
- Object ordering within the PDF file
- Compression details
- Internal identifier assignment

These variations do not affect:
- Visual appearance
- PDF/A compliance
- PDF readers' interpretation

### Future Typst Changes

Typst is under active development. Future versions may produce different output for the same input. To ensure long-term reproducibility:

1. Pin the Typst version in your `Cargo.toml`
2. Store generated PDFs in version control
3. Use containerized builds for CI/CD

## Troubleshooting

### Output Not Deterministic

If output varies between runs:

1. **Check timestamp option**: Ensure `fixed_timestamp` is set
2. **Check Typst version**: Ensure consistent dependency versions
3. **Check for system fonts**: Verify `include_system_fonts(false)` is used
4. **Run tests**: Execute `cargo test --package rtfkit-render-typst determinism`

### Different Output on Different Machines

If output differs across machines:

1. **Verify Typst version**: Both machines must use the same version
2. **Check embedded fonts**: Ensure embedded fonts are used (not system fonts)
3. **Check timestamp**: Ensure the same fixed timestamp is used

### CI/CD Pipeline Issues

For CI/CD determinism:

1. Use a fixed `Cargo.lock` file
2. Pin all dependencies including transitive ones
3. Use containerized builds with fixed base image
4. Set `fixed_timestamp` to a consistent value
